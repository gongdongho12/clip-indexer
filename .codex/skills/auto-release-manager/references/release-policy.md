# Clip Atlas Release Policy

## Branches

- `main` stays release-ready.
- Short-lived feature branches merge into `main` through pull requests.
- `release/vX.Y` is only for hotfix lines.
- `release/auto` is the automatic prerelease branch. A push to this branch should create a new prerelease tag only after formatting, vet, and tests pass.

## Tags

- Stable releases use `vMAJOR.MINOR.PATCH`, for example `v0.2.0`.
- Automatic branch releases use `vX.Y.Z-auto.YYYYMMDD.RUN.SHORTSHA`.
- The automatic tag is based on the latest stable `vX.Y.Z` tag. If none exists, use `v0.1.0` as the base.
- Tags trigger artifact publishing. Branch pushes should not upload release assets directly.

## Release Notes

Release notes are generated by `.github/scripts/release_notes.py` from commit subjects since the previous stable tag.

Preferred commit subjects:

```text
feat: add batch apply preview
fix: keep cache valid after metadata-only changes
ci: publish checksums with release assets
docs: document automatic release branch
```

`feat:` entries appear under `Features`. If no feature commits exist, the notes should still include a `Features` section that states no feature commits were found.

## Version Injection

Release builds inject these package variables in `internal/media` using Go linker flags:

- `version`
- `commit`
- `date`

The CLI should continue to show these values through `clip-indexer --version`.
